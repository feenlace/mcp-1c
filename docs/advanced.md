# Платные версии mcp-1c

Открытая версия mcp-1c дает AI-ассистенту доступ к вашей базе 1С: 9 инструментов, метаданные, поиск по коду, запросы к данным. Для многих задач этого хватает.

Но в ежедневной работе есть то, что базовыми инструментами не решается. Запрос тормозит, в расширениях баг, после обновления платформы что-то отвалилось, нужно переписать модальные вызовы на асинхронные. На каждую такую задачу уходит полдня.

Расширенная версия объединяет инструменты Открытой в 8 консолидированных инструментов с параметром `action` и добавляет анализ кода, генерацию, память по базе и библиотеку шаблонов. Профессиональная версия идет дальше: семантический поиск, граф зависимостей, массовый анализ всей кодовой базы, аудит безопасности, метрики качества, генерация тестов и обработок, навигация по типовым конфигурациям и помощник по обновлению типовых.

## Расширенная версия: 8 консолидированных инструментов

В Расширенной версии возможности Открытой собраны в консолидированные инструменты с параметром `action`: `code_read`, `code_search`, `code_execute` и `system`. К ним добавлены четыре новых инструмента: `code_analyze`, `code_generate`, `memory` и `templates`. Девять отдельных инструментов Открытой версии никуда не деваются, они становятся действиями внутри консолидированных инструментов.

### code_read - чтение кода и метаданных

Объединяет чтение метаданных и кода: дерево объектов, структуру объекта, структуру формы, информацию о конфигурации, исходный код модулей. Действия:

- `metadata_tree` (Открытая) - дерево объектов конфигурации по категориям
- `object_structure` (Открытая) - реквизиты, табличные части, измерения, ресурсы и типы полей объекта
- `form_structure` (Открытая) - элементы, команды и обработчики управляемой формы
- `config_info` (Открытая) - имя, версия, поставщик, режим совместимости конфигурации
- `module_code` (Расширенная) - исходный код модуля объекта, формы, общего модуля или расширения
- `context` (Расширенная) - сжатые метаданные объекта с кэшированием: связанные объекты, регистры, модули, подписки на события
- `resolve` (Расширенная) - резолв имени или GUID объекта в его модули и формы, мост к действию `module_code`
- `get_xsd_schema` (Расширенная) - структурная XSD-схема типа объекта, выведенная из вашей выгрузки. Это не авторитетная XSD 1С, а структура по образцам дампа
- `verify_xml` (Расширенная) - структурная проверка XML против схемы из выгрузки: состав, вложенность, обязательность и типы листьев. Не заменяет авторитетную XSD-валидацию 1С
- `dossier` (Профессиональная) - полное досье объекта одним вызовом (см. раздел Профессиональной)

### code_search - поиск по коду и справочнику

Объединяет полнотекстовый поиск по коду модулей и справочник по синтаксису платформы. Действия:

- `text` (Открытая) - поиск по коду модулей в режимах `smart` (BM25, по умолчанию), `regex`, `exact`. Векторные режимы `semantic` и `hybrid` доступны в Профессиональной редакции при запуске с `--enable-semantic`
- `syntax_help` (Расширенная) - справка по функциям и типам платформы из файлов синтакс-помощника (.hbk). Поиск понимает русскую морфологию: запрос "работа с датами" находит раздел "Работа с датами и временем"

Справочник содержит около 10 000 определений методов, свойств и типов. Данные берутся из .hbk-файлов документации платформы, тех самых, что использует Синтакс-помощник в Конфигураторе. Работает без Конфигуратора, без EDT, без интернета.

### code_execute - выполнение кода и запросов

Объединяет выполнение запросов и проверку их синтаксиса, добавляет выполнение произвольного кода BSL. Действия:

- `query` (Открытая) - выполнить запрос ВЫБРАТЬ/SELECT к данным 1С, безопасно для рабочих баз
- `validate` (Открытая) - проверка синтаксиса запроса без выполнения
- `code` (Расширенная) - выполнить произвольный код BSL в контексте базы 1С

Действие `code` может изменять данные, поэтому инструмент помечен как не доступный только для чтения. Все операции, изменяющие данные, проходят через песочницу с подтверждением (см. раздел "Безопасность").

### system - системные операции

Объединяет чтение журнала регистрации и список подключенных баз, добавляет управление кэшем расширений. Действия:

- `event_log` (Открытая) - чтение журнала регистрации 1С с фильтрами по дате, уровню важности и пользователю
- `list_bases` (Открытая) - список подключенных баз 1С и их статус
- `refresh_extensions` (Расширенная) - принудительно обновить кэш расширений конфигурации (.cfe)

### code_analyze - анализ и проверка кода

Статический анализ модулей и проверка качества кода. Действия:

- `explain` - глубокий статический анализ кода BSL с шестью анализаторами: сложность, ошибки, стиль, производительность, безопасность, совместимость. Структура процедур, зависимости от объектов метаданных, побочные эффекты, экспортные методы, уровень вложенности
- `check` - проверка кода BSL на типичные ошибки и антипаттерны: незакрытые транзакции, запросы в цикле, динамическое выполнение, пустые блоки Исключение, устаревшие методы, мертвый код
- `compatibility` - анализ совместимости с версиями платформы 1С: deprecated-методы, переименования, breaking changes и предложенные замены. Поддерживается матрица совместимости для нескольких версий одновременно

В Профессиональной редакции к инструменту добавляются действия `dependency_graph`, `call_hierarchy`, `arch_boundary`, `find_modules` и `graph_query` (см. раздел Профессиональной).

### code_generate - генерация и оптимизация кода

Генерация кода по описанию на русском языке и рефакторинг. Действия:

- `query` - генерация запросов 1С по текстовому описанию с учетом виртуальных таблиц регистров и реальных метаданных вашей конфигурации
- `print_form` - генерация кода печатных форм: процедура печати, XML-макет табличного документа с областями (Шапка, СтрокаТаблицы, Подвал), регистрация команды печати
- `async_convert` - преобразование синхронных модальных вызовов в асинхронные
- `optimize_query` - статический анализ запросов на 15 антипаттернов производительности: соединения по неиндексированным полям, ВЫБРАТЬ вместо конкретных полей, подзапросы вместо временных таблиц, функции в условиях, отсутствие РАЗРЕШЕННЫЕ, запросы в цикле

Действие `async_convert` переводит конструкции для веб-клиента и мобильного приложения (8.3.18+):

| Было | Стало |
|------|-------|
| `Вопрос(...)` | `ПоказатьВопрос(Оповещение, ...)` |
| `Предупреждение(...)` | `ПоказатьПредупреждение(Оповещение, ...)` |
| `ВвестиСтроку(...)` | `ПоказатьВводСтроки(Оповещение, ...)` |
| `ВвестиЧисло(...)` | `ПоказатьВводЧисла(Оповещение, ...)` |
| `ВвестиДату(...)` | `ПоказатьВводДаты(Оповещение, ...)` |
| `ПоместитьФайл(...)` | `НачатьПомещениеФайла(Оповещение, ...)` |
| `ПолучитьФайл(...)` | `НачатьПолучениеФайла(Оповещение, ...)` |

### memory - память проекта

Долговременная память, привязанная к конкретной базе 1С. AI-ассистент сам сохраняет важный контекст (заметки, решения, договоренности) и поднимает его в следующих сессиях, не теряя нить работы по проекту. Действия:

- `remember` - сохранить заметку, привязанную к текущей базе
- `recall` - найти ранее сохраненные заметки по запросу
- `forget` - удалить заметку

В Расширенной редакции `recall` ищет по ключевым словам. В Профессиональной редакции при запуске с `--enable-semantic` работает как семантический поиск по смыслу (TF-IDF и синонимы 1С).

### templates - библиотека шаблонов кода

Курируемая библиотека переиспользуемых фрагментов BSL и запросов 1С, общая для всех баз. Вы пополняете ее проверенными решениями, AI-ассистент подбирает подходящий шаблон по описанию задачи. Действия:

- `add` - добавить шаблон в библиотеку
- `list` - список доступных шаблонов
- `get` - получить шаблон по идентификатору
- `search` - найти шаблон по описанию задачи
- `remove` - удалить шаблон

В Расширенной редакции `search` ищет по ключевым словам. В Профессиональной редакции при запуске с `--enable-semantic` работает как семантический поиск по смыслу.

### Расширения .cfe

Расширенная версия автоматически обнаруживает установленные расширения конфигурации (.cfe) и включает их модули в `code_search` и `code_read`. Результаты расширений помечаются префиксом `[Расш]`, чтобы было видно, откуда пришел код. Когда в базе 10+ расширений от разных разработчиков, без этого отладка конфликтов превращается в мучение.

```
> Найди все обработчики ПриЗаписи в расширениях
> Покажи код модуля объекта из расширения МоеРасширение
```

### Дополнительные возможности Расширенной

**Мультибазовость.** Одновременная работа с несколькими базами 1С: сравнить структуру метаданных между dev и prod, искать код в нескольких конфигурациях, работать с тестовой и рабочей базой в одном сеансе.

```bash
mcp-1c-advanced --base dev=http://localhost:8081/hs/mcp-1c --base prod=http://localhost:8082/hs/mcp-1c
```

**Работа без Apache/IIS (long polling).** В стандартном режиме mcp-1c требует HTTP-сервис 1С. Long polling переворачивает схему: не mcp-1c стучится в 1С, а расширение в базе само опрашивает Go-сервер за заданиями, выполняет и возвращает результат. Итого расширение в базе плюс бинарник mcp-1c, без публикации HTTP-сервисов.

**Умный контекст.** Сжатие метаданных и выборочная подача. Вместо полного дерева метаданных (для ERP это сотни килобайт) модель получает только релевантную часть. Спрашиваете про РеализациюТоваровУслуг, AI-ассистент получит структуру этого документа и связанных регистров, а не всю конфигурацию. Больше места в контекстном окне, меньше "забывания" в длинных диалогах.

## Профессиональная версия: инструменты

Профессиональная версия включает все инструменты Расширенной и добавляет глубокий анализ всей кодовой базы, архитектурную визуализацию, автогенерацию документации, генерацию тестов и обработок, навигацию по типовым конфигурациям и структурный diff расширений. Большинство возможностей требуют предварительного построения кэша или разовой генерации. CLI-команды и примеры запуска в этом разделе используют бинарник `mcp-1c-pro`.

### Дополнительные действия консолидированных инструментов

Профессиональная редакция расширяет инструменты Расширенной новыми действиями.

**code_read -> dossier.** Полное досье объекта одним вызовом: структура, где используется (used_by), движения по регистрам (movements), связанные объекты (related). Богаче, чем `context`: значения перечислений и граф связей. При отсутствии графа зависимостей возвращается только структура объекта.

**code_search -> семантический поиск.** Поиск по смыслу, а не по тексту. Находит связанный код, даже если названия отличаются. Канонический способ - действие `text` с параметром `mode: "semantic"` (чистый векторный поиск, LSA и Randomized SVD) или `mode: "hybrid"` (комбинация BM25 и векторного поиска). Требует флага `--enable-semantic` при запуске. Прежнее действие `semantic` устарело и будет удалено в одной из будущих мажорных версий.

**code_analyze -> dependency_graph.** Граф зависимостей объектов: визуализация связей между модулями конфигурации. Направление обхода (кто зависит от объекта, от кого зависит объект, оба), глубина, фильтры по типу связи и уровню риска, ограничение по glob-пути. Форматы вывода: текст, JSON, Mermaid. Требует `--dump` и `--build-depgraph`.

**code_analyze -> call_hierarchy.** Иерархия вызовов метода: кто вызывает метод и кого вызывает он сам, трассировка на уровне методов по всей конфигурации. Учитываются квалифицированные вызовы вида `Модуль.Метод()`, вызовы без префикса и динамические не отражаются. Требует граф зависимостей.

**code_analyze -> arch_boundary.** Анализ нарушений границ подсистем (правило ARCH001): обращения к служебным общим модулям одной подсистемы из других подсистем, то есть использование внутренней реализации в обход публичного интерфейса. Помогает поддерживать чистую модульную архитектуру. Работает по дереву подсистем дампа.

**code_analyze -> find_modules.** Поиск модулей по индексу путей выгрузки: по glob-шаблону пути, расширению файла, категории объекта, виду модуля или глубине пути. Помогает ориентироваться в большой конфигурации, не обходя полное дерево метаданных.

**code_analyze -> graph_query.** Структурные запросы к графу зависимостей в формате JSON: ищите узлы и пути по шаблону, а не обходите граф от одного объекта. Примеры: какие документы делают движения по конкретному регистру, общие модули, которые никто не вызывает, кратчайшие пути вызовов между двумя объектами, вызовы в обход границ подсистем.

### bulk_analyze - массовый анализ кодовой базы

Комплексный анализ всей кодовой базы за один проход: антипаттерны, дубли кода, мертвый код, аудит безопасности, метрики качества. Требует предварительного построения кэша командой `--build-bulkanalysis`. Действия:

- `summary` - сводка по всей кодовой базе: количество модулей, находок по категориям, общий балл техдолга
- `findings` - антипаттерны и ошибки кода (30 правил): неоптимальные запросы, утечки ресурсов, устаревшие конструкции
- `duplicates` - дубли кода (MinHash и LSH): похожие фрагменты в разных модулях с процентом совпадения
- `deadcode` - мертвый код: неиспользуемые процедуры, переменные, параметры
- `metrics` - метрики качества: LOC, цикломатическая сложность, глубина вложенности, балл техдолга (0-100)
- `trends` - динамика показателей между запусками анализа

Аудит безопасности (`findings` с категорией `security`, 11 правил SEC) находит SQL-инъекции, небезопасные вызовы и утечки данных. У находок приводятся оценки CVSS (вектор, балл, уровень критичности) рядом со ссылками на CWE и OWASP, а также справочные сопоставления с мерами ГОСТ Р 57580.1 и приказа ФСТЭК России N 21. Это справочная информация для оценки и приоритизации, не сертификация и не подтверждение соответствия. Находки можно подавлять прямо в коде комментарием `// audit:ignore ПРАВИЛО -- причина`, в том числе с указанием срока действия.

### update_helper - сравнение конфигураций и помощник обновления

Сравнение конфигураций (2-way или 3-way) для подготовки к обновлению: что изменилось между вашей доработанной конфигурацией и поставкой вендора. Анализ выполняется асинхронно. Действия:

- `analyze` - поставить сравнение в очередь и вернуть `job_id`
- `summary` - статус задачи (queued, running, done, error) с прогрессом
- `get_report` - итоговый отчет, когда задача завершена

Если задана старая поставка вендора, выполняется 3-way сравнение, иначе 2-way. Форматы отчета: markdown, text, json, html, pdf.

### Архитектурная визуализация

Mermaid-диаграммы трех уровней архитектуры конфигурации: иерархия подсистем и их вложенность, связи документов с регистрами накопления, сведений и бухгалтерии, бизнес-процессы с точками маршрута и BSL-обработчиками. Опционально экспорт в SVG или JSON. Поддерживается фильтрация по подсистеме или категории объектов, ограничение числа узлов, пакетный экспорт всех режимов сразу.

```bash
mcp-1c-pro --build-archviz --arch-viz all --output-dir ./diagrams --dump /path/to/dump
```

В режиме MCP-сервера доступны действия `generate` (построить или обновить кэш диаграмм), `get_subsystems`, `get_doc_reg`, `get_bp`, `get_json`. При ответе свыше 1 MiB сервер сохраняет артефакт в локальный кэш и возвращает manifest с путем, размером и SHA-256, а AI-клиент читает файл напрямую.

### Автогенерация документации

Команда `--build-autodoc` формирует отдельный Markdown-файл для каждого объекта конфигурации с описанием назначения, реквизитов, табличных частей и связей с другими объектами. Использует внешний LLM-провайдер (YandexGPT или GigaChat). Результаты кэшируются: повторный запуск регенерирует только изменившиеся объекты.

```bash
mcp-1c-pro --build-autodoc --dump /path/to/dump --autodoc-out ./docs/auto --autodoc-provider yandexgpt
```

В режиме MCP-сервера доступны действия `generate`, `regenerate`, `summary`, `find`. Автодокументация полностью построена на ответах модели и без API-ключей не запускается.

### Генерация тестов

Команда `--build-testgen` анализирует BSL-модули и формирует scaffold-тесты YAxUnit для экспортированных процедур и функций (правила TG001-TG019), Vanessa-сценарии `.feature` для находок безопасности с критичностью не ниже Medium (правило TG020) и файл `manifest.json` с итогами. Доступна как разовая генерация (`--build-testgen`) или в режиме MCP-сервера (`--enable-testgen`).

```bash
mcp-1c-pro --build-testgen --dump /path/to/dump --testgen-framework auto --testgen-output ./tests
```

Фреймворк выбирается флагом `--testgen-framework`: `yaxunit`, `vanessa`, `both` или `auto`. Флаг `--testgen-llm-disable` генерирует только скелеты тестов без LLM-комментариев и работает без API-ключей. В режиме сервера доступны действия `generate`, `summary`, `findings`, `manifest`.

### Генерация .epf-обработок

Команда `--build-epfgen` формирует Designer-compatible бандлы XML и BSL для всех шести видов внешних обработок 1С: ПечатнаяФорма, ЗаполнениеОбъекта, СозданиеСвязанныхОбъектов, Отчет, ДополнительнаяОбработка, ДополнительныйОтчет. Каждый бандл соответствует контракту БСП и готов к загрузке Конфигуратором. Доступна как разовая генерация (`--build-epfgen`) или в режиме MCP-сервера (`--enable-epfgen`).

```bash
mcp-1c-pro --build-epfgen --epfgen-spec ./bundles.yaml --epfgen-pack
```

Опционально формируются скрипты `pack.cmd` и `pack.sh` для сборки бинарного `.epf`. В config-aware режиме при наличии `Configuration.xml` LLM может предлагать имена реквизитов, согласованные с метаданными целевой конфигурации. Флаг `--epfgen-llm-disable` работает без API-ключей. В режиме сервера доступны действия `generate`, `summary`, `findings`, `manifest`, `validate`.

### Навигация по типовым конфигурациям

Команда `--build-typicalconfigs` определяет семейство типовой конфигурации по сигнатурам метаданных и строит двусторонний индекс "подсистема - объект" для AI-навигации. Распознаются основные линейки: Бухгалтерия предприятия 3.0, Зарплата и управление персоналом 3.1, Управление торговлей 11, Розница, Комплексная автоматизация и ERP. Объекты классифицируются на общие (БСП), специфичные для БП, ЗУП, УТ и Розницы (правила TC001-TC020), поверх классификации детектируются кастомные доработки заказчика.

```bash
mcp-1c-pro --enable-typicalconfigs --base main=https://erp.example.com --dump main=./dump
```

В режиме MCP-сервера (`--enable-typicalconfigs`) AI-ассистент получает действия `detect` (определить семейство), `subsystems` (список подсистем), `lookup` (найти объект), `reverse_lookup` (из объекта в подсистемы) и `extension_points` (точки расширения).

### code_review - ревью расширений

Консолидированный инструмент с двумя действиями для работы с расширениями:

- `review_extension` (CFEDiff) - структурный diff двух версий `.cfe`. Распаковывает оба расширения, разбирает структуру и выдает Markdown-отчет с детальным сравнением BSL-кода, форм и метаданных. Замена ручному diff бинарных `.cfe`, которые показывают только "файл изменен"
- `base_vs_ext` - сверка основной конфигурации с расширением: собственные объекты расширения, переопределенные свойства заимствованных объектов, добавленные реквизиты и табличные части, перехваты `&Вместо` и `&ИзменениеИКонтроль` относительно базы

Доступно как MCP-инструмент и как CLI-подкоманда `review` для CI и pre-commit:

```bash
mcp-1c-pro review base.cfe head.cfe --format markdown -o diff.md
# exit code: 0 = нет различий, 1 = найдены различия, 2 = ошибка
```

Форматы вывода markdown, json, text. Поддерживается чтение как canonical-выгрузки из Конфигуратора (`DESIGNER /DumpCfg`), так и выгрузки `ibcmd config export` (8.3.27+). Готовый шаблон GitHub Actions есть в репозитории mcp-1c-advanced.

### LLM-провайдеры для функций Pro

Четыре функции Профессиональной редакции (автодокументация, генерация тестов, генерация .epf-обработок, навигация по типовым конфигурациям) используют внешний LLM API для обогащения результатов. Поддерживаются два провайдера: YandexGPT (по умолчанию) и GigaChat. Учетные данные читаются из переменных окружения:

```bash
# YandexGPT
export YANDEXGPT_API_KEY=AQVN...
export YANDEXGPT_FOLDER_ID=b1g...

# GigaChat
export GIGACHAT_CLIENT_ID=...
export GIGACHAT_CLIENT_SECRET=...
```

Генерацию тестов, .epf-обработок и навигацию можно запускать и без LLM (флаги `--testgen-llm-disable`, `--epfgen-llm-disable`, `--typicalconfigs-llm-disable`). Автодокументация без API-ключей не работает.

### CI/CD и отчеты

Инструменты Pro рассчитаны на встраивание в конвейер: SARIF 2.1.0 для bulk_analyze, testgen, epfgen и typicalconfigs, отчеты в HTML и PDF, quality gates по порогам. Подкоманда `review` возвращает разные коды выхода для блокировки PR при превышении порога различий.

## Безопасность

**Локальный процесс.** MCP-1C работает на вашей машине. Общается только с вашей базой 1С по локальной сети. Никаких внешних серверов для работы инструментов не требуется.

**Локальные модели.** При использовании Ollama, LM Studio или llama.cpp данные не покидают вашу инфраструктуру. Подходит для закрытых контуров и работы с конфиденциальными данными (зарплата, персональные данные, финансовая отчетность).

**Песочница.** Выполнение произвольного кода BSL (`code_execute` с действием `code`) проходит через песочницу. Все операции, изменяющие данные, требуют явного подтверждения. Действуют ограничения:

- Черный список опасных операций (удаление объектов, транзакции, мета-исполнение, внешние обработки)
- Защита от обхода блокировок через гомоглифы и zero-width символы
- Ограничение размера кода (10 КБ)
- Режим "только чтение" по умолчанию

**Аудит.** Каждое действие пишется в журнал (JSON Lines).

**Офлайн-режим.** Сервер продолжает работать между проверками лицензии, поэтому кратковременная потеря связи с сервером лицензий не прерывает работу. Подходит для объектов с нестабильным подключением.

### AI-клиенты и модели

Все инструменты работают с любой моделью. MCP-1C не привязан к конкретной нейросети, не знает и не определяет, какая модель работает на стороне клиента.

| | Примеры |
|---|---|
| **Локальные модели** | Ollama, LM Studio, llama.cpp - данные не покидают вашу сеть |
| **Облачные сервисы** | Claude, GPT, YandexGPT, GigaChat |
| **IDE с AI** | Cursor, VS Code (Continue, Cline, Copilot), Windsurf, JetBrains IDEs |

Какую модель использовать, решаете вы. Для работы с конфиденциальными данными 1С подходят локальные модели: весь обмен происходит на вашей машине, ничего не уходит наружу.
